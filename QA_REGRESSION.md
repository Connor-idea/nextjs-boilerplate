# CRM 业务回归脚本

## 1. 目的

本文档用于把当前最关键的三条业务链路固化成可重复执行的回归脚本：

- AI 分配闭环
- AI 推客闭环
- 供应商对账闭环

文档同时给出自动化验证命令和手工端到端检查步骤，便于开发、测试和演示前快速确认当前主干能力没有回退。

---

## 2. 环境准备

建议使用如下前置条件：

1. Node 版本尽量对齐 `20.x`。
2. 根目录依赖与 `backend` 依赖都已安装完成。
3. 在项目根目录执行：

```bash
npm run dev:full
```

如果需要分别启动，也可以使用：

```bash
npm run dev
npm run dev:api
```

---

## 3. 自动化回归命令

### 3.1 业务回归集

```bash
npm run test:business
```

该命令当前覆盖：

- `src/lib/crmAdapters.test.js`
  - AI 推客任务构建
  - AI 推客结果回写合并
  - 供应商附件序列化
- `backend/src/crm/crm-state.service.test.ts`
  - AI 分配后端写回
  - 推客快照写回与画像派生
  - 供应商账款同步与通知已读

### 3.2 编译校验

```bash
npm run build
npm run build:api
```

## 4. AI 分配闭环手工脚本

### 4.1 目标

确认线索选择、AI 预分配、确认回写、通知增长和跨页面同步全部成立。

### 4.2 步骤

1. 打开 `销售线索 -> 线索管理`，确认页面路径为 `/sales/leads`。
2. 在列表中勾选至少 `1` 条线索，优先选择当前负责人为 `未分配` 的线索。
3. 点击顶部批量操作区的 `AI智能分配`。
4. 确认页面跳转到 `/sales/leads/ai-assign`，并且当前页能看到与勾选数量一致的 AI 预分配结果。
5. 如需人工干预，可在分配页调整负责人，然后点击 `确认分配`。
6. 确认页面出现成功提示，且不报接口错误。
7. 返回 `线索管理` 页面，确认：
   - 对应线索负责人已更新。
   - 新分配线索的状态从 `新线索` 升级为 `二次分配线索`。
   - 分配日志新增一条 `AI智能分配` 记录。
8. 打开首页，确认未读通知数量增加，连接状态仍为 `connected`。
9. 打开 `销售画像`，确认被分配销售的负载或画像指标出现对应变化。

### 4.3 通过标准

- 分配结果能从分配页回写到线索主池。
- 首页通知数变化与分配动作一致。
- 画像页读取到更新后的后端派生结果。

## 5. AI 推客闭环手工脚本

### 5.1 目标

确认 AI 推客模块能消费共享线索、回写跟进结果，并带动首页与画像页同步刷新。

### 5.2 步骤

1. 打开 `销售线索 -> AI推客`，确认路径为 `/sales/pitch`。
2. 在左侧任务上下文中确认当前线索不是 `未分配` 状态；如无可处理线索，先回到线索页做一次 AI 分配。
3. 在动作区点击 `填写跟进`，打开右侧抽屉。
4. 在抽屉中补充本次跟进所需的备注或信息；如果页面校验要求填写字段，则先补全必填项。
5. 点击 `提交 / 下一条`。
6. 观察提示信息，确认本条线索被处理后自动切到下一条线索，或至少当前列表状态已变化。
7. 返回首页，确认：
   - `待跟进线索` 指标发生合理变化。
   - 未读通知数量按同步逻辑刷新。
8. 打开 `销售画像`，确认对应销售的 `已完成任务数` 或负载状态出现变化。
9. 返回 `线索管理`，确认对应线索的跟进状态、备注或历史已同步回主线索池。

### 5.3 补充检查

可额外执行一次 `退回线索池` 分支检查：

1. 在动作区点击 `退回线索池`。
2. 在确认抽屉中输入要求的确认文本 `退回线索池`。
3. 点击 `执行退回`。
4. 确认当前线索从推客上下文中移出，并在主线索池中体现状态变更。

### 5.4 通过标准

- 推客动作能够回写共享线索主池。
- 首页和画像页消费的是回写后的统一快照，而不是旧静态数据。

## 6. 供应商对账闭环手工脚本

### 6.1 目标

确认财务对账详情的修改会进入共享状态、后端快照与通知系统。

### 6.2 步骤

1. 打开 `财务管理 -> 供应商对账款`，确认路径为 `/finance/supplier`。
2. 在列表中点击任意一条账单进入详情页。
3. 在详情页底部 `账款备注` 区修改备注内容。
4. 观察页面状态，确认出现 `有未保存更改` 提示。
5. 点击底部 `保存`。
6. 确认状态标签从未保存切换为 `已保存`，且页面没有报错。
7. 返回首页，确认：
   - 未读通知数量增加。
   - 首页 `供应商账款` 数量与实际账单总量保持一致。
8. 重新进入同一账单详情，确认刚才编辑的备注仍然存在。

### 6.3 附件序列化补充检查

如需验证附件序列化路径，可追加如下步骤：

1. 在支持上传的附件区域选择一张图片或一个 PDF。
2. 点击 `保存`。
3. 重新打开当前账单，确认附件名称仍可显示。
4. 观察前端控制台与后端日志，确认没有出现 `File` 不可序列化或 JSON 传输失败错误。

### 6.4 通过标准

- 保存动作会进入共享状态与后端快照。
- 通知区能够收到新的财务同步通知。
- 上传后的附件以可序列化元数据形式保留。

## 7. 回归完成标准

当以下条件全部满足时，可判定当前主干通过回归：

1. `npm run test:business` 通过。
2. `npm run build` 通过。
3. `npm run build:api` 通过。
4. AI 分配、AI 推客、供应商对账三条手工闭环脚本均通过。
5. 首页连接状态保持 `connected`，且通知区能反映各链路的写回结果。
